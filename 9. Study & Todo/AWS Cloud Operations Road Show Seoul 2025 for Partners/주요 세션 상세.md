② 실제 파트너가 고객과 함께 활용할 수 있는 진단 툴(AWS Partner Central Assessment Tool) 소개

- 관측성의 중요성은 시스템 복잡도가 증가함에 따라 더욱 커졌으며, 마이크로서비스·컨테이너 환경에서의 통합 모니터링이 핵심 과제임.
Observability 4단계 성숙도 모델	

1단계: 기본 모니터링 (Foundational) – 팀별로 분산된 텔레메트리 수집 중심.

2단계: 중간 모니터링 (Intermediate) – 수집 데이터 분석 및 알람 생성 단계.
→ 과제: 노이즈 많음, 비즈니스 KPI 미정의.

3단계: 고도 관측성 (Advanced) – 상관관계 분석 및 이상 탐지 수행.
→ 과제: SRE 전문성 필요, 여전히 반응형 운영.

4단계: 예측·자동화 관측성 (Proactive) – 자동화·AIOps 단계, 셀프힐링 시스템으로 발전 중.




- 관련 참고: https://aws-observability.github.io/observability-best-practices/guides/observability-maturity-model/

- 5개 영역(Logs, Metrics, Traces, Processes, AI)에 대해 17개 문항을 평가하고, 1~4단계 점수로 고객의 성숙도를 진단.

핵심 인사이트	- 고객과의 관측성 논의를 사고 발생 시점이 아닌 사전적 컨설팅 형태로 전환할 수 있는 계기 제공.
- 진단과 결과 공유 과정을 통해 고객 Pain Point를 명확히 파악하고, 장기 파트너십 구축의 기회 창출.

Deep Dive – Application Performance Monitoring and Log Analysis

구분	내용
발표자 / 주관	AWS Solutions Architect
주제 개요	CloudWatch 기반의 통합 애플리케이션 성능 모니터링(APM) 및 로그 분석 심화 세션
핵심 내용 요약	- CloudWatch 하나로 애플리케이션, 컨테이너, 서버리스(Lambda)까지 전 계층의 모니터링 커버리지 확보 가능.
- APM의 핵심 지표는 요청 수(Volume), 지연시간(Latency), 문제(Fault), 에러(Error) 네 가지.
- 기존 X-Ray의 비용·제한점을 OpenTelemetry(ADOT) + CloudWatch Application Signals 조합으로 개선.
- Application Signals는 애플리케이션의 모든 시그널(Span, Event)을 수집·분석하며, SDK 기반 또는 ADOT Distro로 손쉽게 적용 가능.

구성 예시 (ECS / EKS / Lambda)
- ECS: adot-autoinstrumentation(init), cloudwatch-agent, app 3개 컨테이너 구성으로 자동 계측 가능.
- EKS: CloudWatch Observability Add-on 설치로 간소화된 모니터링 구축.
- Lambda: Instrumentation Layer 추가를 통한 성능 추적 및 피드백 수집 지원.
새로운 기능 및 주요 포인트	- Application Map (신규 출시) : 주요 애플리케이션 간 호출 관계를 시각화
- Centralization : Multi-Region/Account 로그를 중앙 CloudWatch로 통합 수집 및 암호화(KMS 지원). Log Insights에서 @aws.account, @aws.region으로 필터링 가능.
- Field Index : 로그 인덱싱을 통해 검색 속도 향상 및 비용 절감 (로그 그룹당 20개 무료).
- Transformation : 로그 패턴을 Grok 등으로 변환하여 구조화.
- AI 기능 : Query Generator(자동 쿼리 생성), Log Summarizer(로그 요약) 제공으로 운영 효율 극대화.
핵심 인사이트	- CloudWatch는 단순 메트릭 수집 도구를 넘어 End-to-End Observability 플랫폼으로 진화 중.
- OpenTelemetry와의 통합으로 벤더 종속 없는 계측 구조 구축 가능.
- 로그 중앙화, 인덱싱, AI 분석 기능으로 운영 효율성과 분석 속도가 크게 향상됨.

Scaling Your Business with AWS Cloud Governance Services

구분	내용
발표자 / 주관	AWS Governance Team
주제 개요	AWS Control Tower, Service Catalog, Organizations를 활용한 멀티계정 거버넌스 구축 및 자동화 전략 소개
핵심 내용 요약	- Cloud Governance란 규칙(rule), 도구(tools), 프로세스(process), 보고(report)를 포괄하며, 고객이 비즈니스 목표를 보안·운영·컴플라이언스·비용 관점에서 달성할 수 있도록 돕는 체계를 의미.

1️⃣ Multi-Account 전략의 중요성
- AWS 환경을 조직 단위로 구조화하는 것이 핵심. (참고: Organizing Your AWS Environment Whitepaper)
- 멀티계정 전략은 보안, 청구, 권한, 거버넌스를 분리해 관리 효율성과 규정 준수성을 강화.

2️⃣ AWS Control Tower – 손쉬운 랜딩존 구축
- 콘솔 또는 API를 통해 Landing Zone 자동 생성.
- 계정(Account) 프로비저닝 자동화.
- 700개 이상의 통합 거버넌스 컨트롤을 관리할 수 있는 API 제공.

3️⃣ Control Catalog (Control Tower의 거버넌스 핵심)
- Preventive Controls: 사전 예방 (잘못된 리소스 생성 차단)
- Detective Controls: 감지 (규정 위반 탐지 및 보고)
- Proactive Controls: 선제적 대응 (위반 방지 자동화)

4️⃣ AWS Config – 거버넌스의 중심 서비스
- 단순한 규칙(rule) 엔진이 아니라, TRACK → ACCESS → REMEDIATE → AUDIT의 전체 사이클을 지원.
- 리소스 변경 사항(“누가”, “언제”, “무엇을”)을 추적 가능.
- 예: 1,100개 계정의 EKS 버전 관리(EOS 대응) 시 전체 현황을 한눈에 파악 가능.

5️⃣ AWS Audit Manager
- 규정준수(Audit) 및 생성형 AI 프레임워크 관리 기능 제공 (딜로이트와 협업).

6️⃣ AWS Cloud Adoption Framework (CAF)
- 조직이 클라우드 도입 시 거버넌스, 운영, 보안, 인력, 비용 관점에서 성숙도를 진단하고 로드맵을 수립하는 가이드라인.
- 참고: AWS Cloud Adoption Framework
핵심 인사이트	- Control Tower + Config + Audit Manager 조합은 AWS 거버넌스의 중심축.
- 대규모 Multi-Account 환경에서도 정책 일관성을 유지하고, 보안·비용·운영을 자동화할 수 있음.
- 거버넌스는 단순 제약이 아니라 운영 효율성과 규정 준수의 균형을 위한 관리 체계로 접근해야 함.

핵심 내용 요약	- 엔터프라이즈 환경에서의 관측성 중앙화를 주제로, 다계정·다리전 구조에서 통합 가시성을 구축하는 방법을 설명.

- 멀티 어카운트 환경에서는 팀별·비즈니스별 도구 파편화로 중복 투자 발생.
- AWS 네이티브 기능(CloudWatch, CloudFormation, Config 등)을 통해 중앙 거버넌스 조직에서 통합된 Observability 대시보드 구축 가능.
- 실시간성(Real-time)을 높일수록 비용이 증가하므로, 비용 효율성과 관리 편의성의 균형이 중요함.

2️⃣ CloudWatch Cross-Account 설정 및 확장성
- CloudWatch 콘솔의 Settings > Resource-to-Link Accounts 기능으로 간단히 연결 가능.
- CloudFormation 템플릿을 통해 소스 어카운트 자동 등록 지원.
- 단일 모니터링 어카운트에서 최대 10만 개 소스 어카운트 관측 가능, 데이터 전송 없이 연결되어 추가 DTU 비용 없음.

3️⃣ 대시보드 구성 및 리전 제약
- 하나의 대시보드에서 여러 어카운트의 로그·메트릭은 실시간 조회 가능.
- 단, 서로 다른 리전의 데이터 실시간 통합 시각화는 현재 미지원.

4️⃣ 트레이스 통합 구현 방법
- 다계정 트레이스 통합 시 SDK 레벨에서 TraceID / ParentID 공유 로직 수정 필요.
- 단순 자동 계측(Auto Instrumentation)만으로는 트레이스 연속성이 끊길 수 있음.

5️⃣ 멀티 클라우드 통합 구성 사례
- 고객은 인하우스에 오픈소스 기반 Observability 플랫폼을 구축하고, AWS의 모니터링 어카운트를 통해 데이터를 중간 집계(Aggregation).
- 데이터 트랜스폼 비용을 최소화하면서 멀티/하이브리드 클라우드 환경 통합 완료.
핵심 인사이트	- 중앙화된 Observability는 조직 전체의 데이터 일관성과 통합 분석 역량을 강화함.
- AWS 네이티브만으로도 높은 수준의 통합 모니터링을 구현할 수 있으며, 비용 효율적임.
- SDK 수준의 트레이스 표준화와 업저버빌리티 스펙 정의가 Cross-Account 환경의 완전한 가시성 확보의 핵심임.

Agentic AI를 통한 클라우드 거버넌스의 진화 – Amazon Q와 AWS MCP의 만남

구분	내용
발표자 / 주관	AWS AI Strategy Team
주제 개요	Amazon Q Developer와 AWS MCP를 결합하여 Agentic AI 기반 클라우드 거버넌스 자동화를 구현하는 방안 소개
핵심 내용 요약	- Agentic AI란?
- 단순 LLM(대화형 모델)을 넘어 목표 중심으로 스스로 사고·실행·중단까지 수행하는 자율형 AI.
- 핵심 특징 두 가지:
1️⃣ Stateful Memory – 세션 기억을 기반으로 지속적 맥락 인식 및 학습 가능.
2️⃣ Specialized Agent – 특정 전문 분야(예: SRE, DevOps, 보안)에 특화된 역할 수행 가능.

1️⃣ 협업 모델의 진화: Agentic AI + 인간
- 개발자와 AI가 전략적 의사결정과 실행을 분담하는 구조로 전환.
- “Q Developer × Kiro” 조합을 통해 개발 생산성과 코드 품질 향상.
- AI가 스펙 기반(Spec-Driven)으로 설계~테스트까지 일관성 유지.

2️⃣ Kiro의 역할 (Spec-Driven Development IDE)
- 요구사항 → 설계 → 구현 → 테스트까지 연결하는 SDD(Spec-Driven Development) 기반 구조.
- 단순 코드 생성 도구가 아닌 운영·배포·품질·자동화까지 포괄하는 AI IDE.

3️⃣ Amazon Q CLI + MCP 구조
- 사용자 입력(자연어) → Q CLI → LLM 해석 → MCP Client 호출.
- MCP(Multi Context Protocol)는 Q Developer가 AWS 서비스 컨텍스트와 연결되어 정보를 조회·분석하는 표준 인터페이스.

4️⃣ 주요 MCP 예시
- CloudWatch MCP: 실시간 모니터링, 자동 분석, 통합 관리, 비용 효율성 제공.
- CloudTrail MCP: 이벤트 기록 및 변경 이력 기반 자동 분석 지원.
- 두 MCP를 결합하면 프롬프트 한 줄로 트러블슈팅 및 원인 분석 자동화 가능.

5️⃣ Custom Agent 활용
- “너는 DevOps야”, “너는 SRE야”와 같이 역할별 특화된 Agent 생성 가능.
- AI 동료처럼 협업하며 조직별 워크플로우 자동화 실현.

6️⃣ Amazon Q Developer의 비전
- 조직 내 AI 운영체제로 발전 중.
- 반복적·수동적 업무를 대폭 줄이고, 운영·거버넌스·자동화 전반의 효율성 극대화.
핵심 인사이트	- AWS는 “Agentic AI + MCP + Q Developer”를 통해 **실행 가능한 AI 기반 클라우드 운영 체계(AIOps → AutoOps)**로 발전 중.
- Kiro 기반 Spec-Driven 개발은 DevOps, IaC, CI/CD 자동화에도 확장 가능.
- 향후 AI가 단순 분석을 넘어 운영 정책·코드·거버넌스까지 자율적으로 관리하는 시대로 진입 중.
